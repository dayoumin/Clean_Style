테스트 설계 플레이북

성향 테스트를 처음부터 만들 때 반드시 알아야 할 것들 — Clean Style 프로젝트에서 얻은 교훈

1 설계 비전을 먼저 정한다

점수 체계, 문항 설계, UI 모두 비전에서 파생됩니다. 비전 없이 만들면 나중에 전체를 뒤집어야 합니다.

결정 항목Clean Style의 선택다른 가능한 선택
목적 긍정적 자기발견 — "나쁜 점수" 없음 진단/평가형, 순위형
이론 기반 조직윤리 학술 이론 3축 MBTI, Big Five, 자체 모델 등
결과 톤 강점 중심 서술 약점 지적, 균형 보고
동기 부여 발견 → 유지 → 행동 변화 경고 → 교정
핵심: "이 테스트의 목적이 무엇인가?"를 한 문장으로 정의하고, 모든 후속 결정이 이 문장에 부합하는지 검증합니다.

2 축 구조를 설계한다

양극(bipolar) vs 단극(unipolar)

양극 (내부 계산용)

하나의 축에 +1/-1로 점수 부여.
예: principle: +1(원칙) ↔ principle: -1(유연)
유형 판정은 순점수 부호로 결정 (양수=원칙, 음수=유연).

단극 (차트 표시용)

양극의 +1과 -1을 분리하여 6개 독립 축으로 표시.
원칙(+1 합산), 유연(|-1| 합산) 각각 0~max.
레이더 차트에서 6각형으로 시각화.

함정: 양극을 단극으로 쪼개면 한 쌍의 양쪽이 동시에 낮아질 수 있습니다. 이것이 "축별 최소 참여" 문제의 원인입니다. (아래 Step 4에서 해결)

축 수 × 문항 수 가이드

축쌍 수권장 최소 문항이유
2축쌍10문항축당 5문항 → 변별력 최소선
3축쌍15문항축당 5문항 × 3 (Clean Style 현재)
3축쌍 (개선)18문항축당 6문항 → 안정성 향상
4축쌍+20문항+많아지면 사용자 이탈 주의
문항 수 증가의 핵심 이점: 강제 참여 문항을 자연스럽게 확보하기 쉬워짐. 15문항일 때는 의도적으로 매핑을 추가해야 했지만, 18문항이면 설계 단계에서 배분 가능.

3 점수 표시를 설계한다 (√ 커브)

원시 점수(raw)를 차트 표시 점수로 변환할 때, 테스트 목적에 맞는 커브를 선택합니다.

왜 선형(linear)이 아닌가

대립축 구조에서 선형 매핑을 쓰면 한쪽이 높을 때 반대쪽이 0~1점으로 표시됩니다. "원칙 0점"은 실제로는 "유연 성향이 강하다"는 뜻이지만, 사용자는 "원칙이 없다"로 읽습니다.

제곱근(√) 커브

score = round(MIN + √(raw / max) × (MAX − MIN))
보정설명효과
바닥값 MIN=1 원시 0점이어도 최소 1점 표시 "0점 = 없다"로 오독 방지
√ 커브 중·상위 점수를 넓게 분포 대부분 3~5점 → 긍정적 인상
raw %선형√ 커브x²
0%111
25%231
50%342
75%443
100%555
참고: log 커브는 base에 따라 값이 크게 달라져 단순 비교가 어렵습니다. 일반적으로 √보다 상위로 끌어올리지만 축 간 차이가 거의 사라집니다.
커브 선택 기준:
  • 긍정적 자기발견 목적 → √ (중간을 올림, 차이 유지)
  • 진단/평가 목적 → 선형 (정직한 표현)
  • 게이미피케이션 목적 → log (초반 빠른 성장감, 단 base 선택에 주의)
  • √ 채택 이유: 선형보다 긍정적이면서 축 간 차이가 살아있는 균형점

4 점수 매핑 시 함정을 피한다

함정 1: 축별 최소 참여 미보장
각 선택지가 3축 중 1~2축에만 점수를 주면, 특정 축쌍에 아예 점수가 안 쌓이는 응답 조합이 가능합니다. 레이더 차트에서 한 축쌍의 양쪽이 모두 바닥(1점)으로 표시됩니다.
✅ 해결: "강제 참여 문항"(4개 선택지 모두 해당 축에 점수 부여)을 축쌍당 최소 3개 확보. 검증 도구: npx tsx scripts/check-axis-coverage.ts
함정 2: 시각화 레이어를 검증 안 함
유형 판정(3축 순점수 부호)은 정상이어도, 6축 개별 표시(레이더 차트)에서 문제가 생길 수 있습니다. 로직 레이어만 테스트하면 시각적 문제를 놓칩니다.
✅ 해결: 시뮬레이션에서 유형 분포뿐 아니라 6축 개별 표시 점수도 검증. "무관 선택지만 고른 극단 시나리오"를 시뮬레이션에 포함.
함정 3: 사회적 바람직성 편향
A선택지가 항상 "바른 답"으로 보이면 응답이 편향됩니다. 특히 청렴·윤리 주제에서 심함.
✅ 해결: 선택지 순서를 seededShuffle로 랜덤화. 내용 자체의 바람직성도 매칭되도록 문항 리뷰.
함정 4: 대립축인데 둘 다 높게 표시
√ 커브는 중간 점수를 끌어올리므로, 대립축 양쪽이 모두 4~5점으로 표시될 수 있습니다. "독립 4점, 협력 4점"이면 대립축의 대비가 흐려집니다.
⚠️ 의도적 허용: "독립성도 있고 협력도 한다"로 긍정적 해석 가능. 설계 비전(긍정적 인식)에 부합. 유형 판정은 원시 점수 기반이라 영향 없음.

5 검증 체크리스트

문항 추가·수정 시 반드시 확인. 로직 + 시각화 양쪽을 모두 검증합니다.

#항목기준검증 도구
1 축별 ±1 기회 균형 각 축쌍 양방향 차이 ≤ 3 questions-simulation.ts
2 유형 분포 전 유형 출현율 5% 이상 questions-simulation.ts (100K)
3 사회적 바람직성 특정 선택지가 "정답"으로 안 보이게 question-review-prompt.html
4 축별 최소 참여 강제 참여 문항 ≥ 3개/축쌍 check-axis-coverage.ts
5 차트 시각 검증 축쌍 양쪽 모두 바닥 불가 check-axis-coverage.ts
6 3축 동시 점수 제한 선택지당 최대 3축 (현재 11건) questions-simulation.ts
7 DISPLAY_MAX 적정성 √ 커브 적용 시 차트 60~100% 채움 수동 확인

7 새 테스트를 만들 때 순서

  1. 비전 정의 — 목적·톤·동기부여 방식 결정
  2. 축 설계 — 학술 근거 확보, 양극/단극 결정
  3. 문항 설계 — 축당 5+ 문항, 강제 참여 3+/축쌍 의도적 배치
  4. 점수 매핑 — 각 선택지에 1~3축 점수, 균형 확인
  5. 커브 선택 — 목적에 맞는 정규화 (√, 선형, log)
  6. 시뮬레이션 — 100K 랜덤, 유형 분포·균형·강제 참여 검증
  7. 시각 검증 — 극단 시나리오에서 차트가 정상적으로 보이는지
  8. 사회적 바람직성 리뷰 — 선택지 순서 랜덤화 + 내용 매칭
가장 흔한 실수: 1~4까지만 하고 5~8을 건너뛰는 것. 로직이 맞아도 시각화·사용자 경험에서 문제가 생깁니다. Clean Style에서 √ 커브와 강제 참여 모두 시각 검증 단계에서 발견되었습니다.
